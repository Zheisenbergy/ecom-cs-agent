# 详细项目文档

## 1. 项目定位

这个项目不是要做一个“可上线的电商客服系统”，而是要做一个“适合学习和后训练的 agentic 原型”。

一句话概括：

- 用一个受限工具环境，训练 Qwen 4B 学会在电商客服场景里完成任务闭环。

这里的“任务闭环”比“聊天轮数”更重要。

## 2. 为什么不把重点放在长会话

很多人一开始做客服项目，会直觉上把问题理解成：

- 做一个很像 ChatGPT 的多轮聊天机器人
- 让它一直记住前面的上下文
- 维护 session 和长期 memory

但这对你的目标并不最优。

因为你要做的是：

- agentic 后训练

而不是：

- 长期记忆产品工程

如果目标是 agentic 后训练，你最应该关注的是：

- 模型能不能判断现在该做什么
- 模型能不能把参数填对
- 模型能不能在 observation 后收敛
- 模型能不能在不该回答时拒答或转人工

这些都属于“任务级能力”，不是“长期聊天能力”。

## 3. 最合理的抽象：Episode

### 3.1 什么是 episode

一次完整任务，从用户发起问题开始，到模型给出最终可交付结果结束，这就是一个 episode。

例如：

```text
用户：我的订单到哪了？
助手：请提供订单号。
用户：A1001
助手：订单 A1001 当前在运输中，承运商是中通，最新节点为已到达杭州转运中心。
```

这一整段才算一条完整数据。

### 3.2 为什么 episode 比 session 更适合这里

`session` 这个词更偏产品系统：

- 跨进程
- 跨页面
- 长期保存
- 解决用户体验问题

`episode` 更偏训练和 agent：

- 一次任务闭环
- 有开始和结束
- 适合做标注
- 适合做评测

所以本项目更适合用 `episode` 来做基本单位。

## 4. 当前系统解决了什么痛点

### 4.1 不把业务事实训练进模型

商品属性、订单状态、物流信息、售后规则都不应该靠模型记忆。

原因：

- 这些信息会变
- 每个店铺都不一样
- 训练它们成本高且浪费
- 容易过时和幻觉

更合理的做法是：

- 把它们放在工具或数据库里
- 模型只学会如何调用

### 4.2 把“问答”拆成动作决策

真实客服问题不是只有“回答内容”这一步，前面还有：

- 能不能直接回答
- 需不需要用户补信息
- 要不要调工具
- 是否该转人工

如果只训练“输入一句话，输出一句话”，模型很难学稳这些动作边界。

### 4.3 让训练数据更接近真实任务

按 episode 存数据后，样本天然包含：

- ask_user
- tool_call
- observation
- final_answer
- handoff

这比只有问答对更适合训练 agent。

## 5. 当前系统结构

### 5.1 Router

职责：

- 识别意图
- 判断 route
- 判断是否缺槽位
- 抽取 `order_id` / `product_id`

### 5.2 Internal Tools

职责：

- 提供结构化环境 observation

当前 mock 工具包括：

- 商品查询
- 政策查询
- 订单状态查询
- 物流状态查询

### 5.3 Answer

职责：

- 缺参时输出 ask_user
- 高风险时输出 handoff
- 拿到 tool result 后输出 grounded answer

### 5.4 Orchestrator

职责：

- 管理 episode state
- 运行单步 turn
- 运行完整 episode
- 生成 trace 和 episode record

### 5.5 现在的状态不是“完整历史”，而是“任务状态”

这是一个很关键的点。

当前系统不会在每一轮都把完整历史 transcript 原样拿来重推，而是把与任务推进最相关的信息压缩到 `EpisodeState` 里。

新的状态设计重点是：

- `current_task`
- `recent_queries`

其中 `current_task` 表示“当前正在执行的任务语义”，它会保存：

- 原始任务问句 `task_query`
- 当前任务意图 `intent`
- 计划调用的工具 `planned_tool_name`
- 已有参数 `planned_arguments`
- 缺失槽位 `missing_slots`

这样设计的目的不是模拟真实聊天产品，而是让 agent 训练和评测更稳定。

## 6. 一条完整数据到底怎么算

建议你始终记住这个标准：

一条完整数据 = 一个完整 episode。

### 6.1 直接回答型

```json
{
  "episode_id": "ep-direct-001",
  "query": "你们支持七天无理由吗？"
}
```

### 6.2 单工具闭环型

```json
{
  "episode_id": "ep-tool-001",
  "query": "帮我查订单 A1001 到哪了"
}
```

### 6.3 澄清后闭环型

```json
{
  "episode_id": "ep-clarify-001",
  "turns": [
    {"query": "我的订单到哪了？"},
    {"query": "A1001"}
  ]
}
```

### 6.4 转人工型

```json
{
  "episode_id": "ep-handoff-001",
  "query": "我要投诉商家并申请赔偿"
}
```

## 7. 数据集应该怎么构造

### 7.1 第一阶段最值得做的 4 类

- direct
- ask_user 后完成
- 单工具完成
- handoff

### 7.2 第二阶段再补的

- 多工具串联

例如：

- 先查订单得到 `product_id`
- 再查商品材质
- 再查退货政策

## 8. 如果你是新手，建议先看什么

如果你现在对训练、LoRA、AutoDL、LLaMA-Factory 这些概念还不熟，建议直接先看：

- [新手训练指南](/Users/zheisenbergy/code/agent/ecom-cs-agent/docs/新手训练指南.md)

这份文档会更具体地解释：

- 训练和评测分别在做什么

- 训练前要准备多少数据
- 现在仓库里的数据够不够
- 作为小白第一轮最合理的目标是什么
- 最后合并回答

这会更像真正的 agentic 任务。

当前项目已经做了第一版受控多工具链，典型例子是：

- `我订单 A1001 买的那件衣服是什么材质，能退吗？`

系统会：

1. 调 `get_order_status`
2. 从 observation 中抽取 `product_id`
3. 调 `get_product_info`
4. 调 `get_policy`
5. 合成回答

## 8. 为什么现在不必急着做 RAG 和 Web

因为当前项目最重要的问题还不是“知识不够”，而是“动作不稳”。

如果连：

- ask_user
- tool_call
- handoff
- grounded final answer

都还没训练稳，过早引入 RAG 和 Web 只会把系统复杂度拉高。

正确顺序应该是：

1. 先把受限环境内的 agent loop 跑稳
2. 先有 episode 级训练与评测数据
3. 再考虑引入新 route 和新工具

## 9. 后续如果要加 RAG / Web 怎么办

不需要推翻当前设计。

只要新增：

- route label
- tool schema
- 对应 episode 数据
- 对应评测集

也就是说，当前 episode 设计是可扩展的。

## 11. 当前方案是不是最优

没有绝对最优，但对你当前目标来说，这是最稳、最清晰、最适合学习的一版。

如果后面你想把项目做得更强，最合理的升级方向不是先上 RAG，而是：

- 多工具串联
- 更复杂的 episode
- 更严格的评测
- 再接 Qwen 4B 的 SFT

## 12. 为什么这套设计更接近工业型 Agent

很多工业系统并不会把 agent 做成“完整历史自由聊天 + 完全自由规划”。

更常见的做法其实是把系统拆成三层：

1. 原始对话历史
2. 任务状态
3. 结构化执行轨迹

你当前项目已经在往这个方向收敛：

- 对话原文保留在 episode turn 里
- 当前任务语义保留在 `current_task`
- 工具调用与 observation 保留在 trace 里

这样设计的原因不是保守，而是因为工业界更在意：

- 能不能回放
- 能不能复现
- 能不能定位错误来自 route、tool 还是 answer
- 能不能把 trace 直接转成训练样本

如果完全依赖“每轮都拿完整历史让模型自由决定”，会有几个典型问题：

- 很难做精确回归测试
- 很难抽出稳定的 router / answer 监督数据
- 很难分析某次失败到底是缺槽位、参数错、工具错，还是回答错

所以工业型 agent 往往不是“让模型统治一切”，而是：

- 用模型做决策
- 用状态结构做约束
- 用 trace 做观测和评测

这和研究型 demo agent 的差别很大。

研究型系统更关注：

- 自由规划能力
- 长链推理
- 更开放的行为探索

工业型系统更关注：

- 稳定
- 可控
- 可评测
- 可训练

你的项目目标本来就是 agentic 后训练，所以更适合工业型设计，而不是自由聊天式设计。
