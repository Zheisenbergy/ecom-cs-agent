# 数据构建策略

## 1. 先说结论

对当前这个项目，不建议一开始采用“像多模态项目那样大量爬图片，再让模型脑补商品信息”的造数方式。

原因很直接：

- 当前项目是纯文本 CLI agent，不是图文 VQA 项目
- 你现在要训练的是 `route / ask_user / tool_call / grounded answer`
- 不是训练模型做视觉理解
- 也不是训练模型记住海量商品事实

所以当前阶段最合理的数据路线不是：

- 图片驱动 synthetic data

而是：

- 结构化环境驱动的数据构造
- 模板骨架 + 大模型改写
- 规则生成 + 人工抽检
- 少量高质量、多分布覆盖的 episode 数据

一句话概括：

- 对这个项目，最好的方法不是“自由脑补商品世界”，而是“先定义世界，再围绕这个世界造任务”。

## 2. 什么叫“先定义世界，再造任务”

当前项目里真正稳定的“世界状态”是这些结构化数据：

- 商品信息
- 店铺规则
- 订单信息
- 物流信息

模型不是去背这些事实，而是：

- 学会什么时候查
- 学会查什么
- 学会拿到 observation 后怎么答

所以造数据的正确顺序应该是：

1. 先构造一批结构化 mock 数据
2. 再围绕这些数据生成用户问题
3. 再生成对应的 route / tool / answer / episode trace
4. 最后人工抽查一小部分高价值样本

而不是：

1. 先让大模型自由生成一堆商品和订单故事
2. 再试图从这些故事里反推工具调用标签

后者很容易造成：

- 字段不一致
- 事实漂移
- 工具参数对不上
- 答案看起来合理但不 grounded

## 3. 当前项目最推荐的 4 种造数方法

## 3.1 Schema-driven 规则造数

这是第一优先级。

做法：

- 先定义商品、订单、规则、物流的 JSON schema
- 再按 schema 扩写一批 mock 数据
- 基于这些数据程序化生成 query、tool arguments、gold answer

优点：

- 数据一定和工具环境一致
- 非常适合 agentic 后训练
- 容易做 hard case
- 容易做离线评测

缺点：

- 语言表面形式会比较单一

所以它最适合用来做：

- gold eval
- tool trace 教师数据
- 第一批 router / answer SFT 数据

## 3.2 模板骨架 + LLM 改写

这是第二优先级，也是当前项目最实用的方法。

做法：

- 先人工定义一批 query 骨架
- 骨架里只保留意图、槽位、动作结构
- 再让大模型做用户表达改写

例如同一个意图：

- `帮我查订单 A1001 到哪了`
- `A1001 现在发到哪了`
- `我这单 A1001 物流进度怎么样`
- `订单 A1001 还没到吗`

优点：

- 比纯模板自然很多
- 仍然能保持标签稳定
- 很适合扩增 router 数据

缺点：

- 如果不做约束，模型会改写出超出当前工具能力的问题

所以要注意：

- 改写只改表达，不改任务语义
- 改写后仍然要回到结构化环境里校验

## 3.3 Self-play 多轮补槽造数

这是第三优先级，用来补多轮 episode。

做法：

- 先定义一个“缺槽位”的初始 query
- 再定义 agent 应该 ask_user 什么
- 然后生成用户第二轮补充
- 最后跑完整 episode

例如：

```text
用户：我的订单到哪了？
助手：请提供订单号。
用户：A1001
助手：返回最终物流回答
```

优点：

- 非常适合训练 `ask_user`
- 能补足 episode 数据，而不只是单轮 query

缺点：

- 如果完全让模型自由 self-play，容易出现不自然或越界的补充信息

更好的做法是：

- 首轮 query 用模板或改写生成
- 第二轮补充信息用规则控制
- 最后答案由环境 trace 自动导出

## 3.4 Hard case 定向造数

这是第四优先级，但对 eval 非常重要。

这类数据不一定多，但必须有。

包括：

- 缺订单号
- 缺商品 ID
- 订单不存在
- 商品不存在
- 同时问属性和规则
- 有订单语境但问法很绕
- 应该 handoff 的投诉类问题
- 容易误判成 direct 的短问句

优点：

- 能明显提升边界稳定性
- 对 eval 价值远高于普通样本

这类数据尤其适合放进：

- dev eval
- regression eval

## 4. 当前阶段不推荐的造数方式

## 4.1 纯自由生成商品世界

不推荐做法：

- 让大模型一次性自由生成几千个商品、订单、政策、问答

问题：

- 看起来很多，实际很脆
- 字段容易飘
- 工具调用标签很难严格对应
- 训练后模型会学到“编故事式回答”

## 4.2 图片爬虫 + 模型脑补商品属性

这个方法不是绝对不好，但它更适合：

- 多模态检索
- 图文问答
- VLM tool use

而不太适合当前这个纯文本项目。

当前项目如果这样做，会出现两个偏移：

- 数据重心跑到商品内容构造，不是 agent 行为构造
- 你会花大量时间在“图像数据质量”上，而不是“episode 质量”上

所以现阶段没必要。

## 4.3 完全依赖单一 Prompt 批量生成

例如：

- 给大模型一个 prompt，让它一次性吐 5000 条 tool-call 数据

这类方式常见问题：

- 分布塌缩
- 表达重复
- 工具比例失衡
- handoff 样本太少
- ask_user 样本不真实

可以用，但不能作为主方法。

更合理的定位是：

- 只把它当“改写器”或“扩写器”
- 不把它当“真值构造器”

## 5. 最合理的数据来源组合

对你这个项目，我建议用下面这个组合：

### 第一层：人工定义环境

包括：

- 商品 JSON
- 订单 JSON
- 规则 JSON
- 物流 JSON

这是唯一事实源。

### 第二层：规则生成任务骨架

包括：

- direct
- product_lookup
- policy_lookup
- logistics_status
- order_status
- order_product_lookup
- order_policy_lookup
- order_product_policy_lookup
- handoff

### 第三层：LLM 做表达改写

只改：

- 用户问法
- 口语化表达
- 轻微扰动

不改：

- 真值
- 工具链
- 目标意图

### 第四层：规则环境自动出 teacher trace

通过当前项目已有的：

- `run`
- `trace`
- `export-router-sft`
- `export-answer-sft`

自动产出训练数据。

### 第五层：人工抽查高价值样本

重点抽查：

- 多工具 chain
- ask_user
- handoff
- not_found
- 边界歧义样本

## 6. 对当前项目的推荐配比

如果你准备做第一轮可用训练，推荐大概这样分：

- 40% 单工具闭环
- 20% ask_user 后闭环
- 15% direct
- 10% handoff
- 15% 多工具 chain

同时还要单独保留一份 hard eval：

- 缺槽位
- not_found
- 模糊问法
- 投诉升级
- 多工具串联

注意：

- 训练集里 handoff 不要太少
- eval 里 hard case 比例要明显高于 train

## 7. 如果以后加 RAG / Web，数据方法怎么变

如果以后真的要加 RAG / Web，不需要推翻当前方法。

只需要新增：

- 新 route
- 新 tool schema
- 新 observation 类型
- 新 episode 模板

这时再增加：

- 基于检索结果的 grounded answer 数据
- 基于网页摘要的 answer 数据
- route 到 RAG / Web 的判断数据

也就是说：

- 当前阶段先做“结构化工具环境造数”
- 后续阶段再做“检索增强环境造数”

## 8. 你现在最应该怎么做

不是去爬图片，也不是去一次性造几万条自由数据。

而是按这个顺序：

1. 先把 mock 商品、订单、规则、物流扩充到足够覆盖主要意图
2. 再补 query 模板
3. 再用 LLM 做改写扩增
4. 用规则系统跑 episode trace
5. 导出 router / answer 数据
6. 做一版小而硬的 eval 集
7. 开始第一轮 LoRA SFT

## 9. 最终建议

如果只给一个建议，那就是：

- 不要把“造数据”的重点放在“生成商品事实”
- 要把重点放在“生成可验证的任务闭环”

因为你这个项目的核心不是商品百科，而是：

- agent 在一个受限环境里能不能稳定做对动作
- 能不能在 observation 后输出 grounded answer
